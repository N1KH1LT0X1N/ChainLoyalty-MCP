# Security Policy\n\n## Supported Versions\n\n| Version | Supported          |\n|---------|-------------------|\n| 1.0.x   | :white_check_mark: |\n| < 1.0   | :x:               |\n\n## Reporting a Vulnerability\n\nWe take the security of ChainLoyalty MCP Server seriously. If you discover a security vulnerability, please report it responsibly.\n\n### How to Report\n\n**Please do NOT report security vulnerabilities through public GitHub issues.**\n\nInstead, please send an email to: **security@chainloyalty.com**\n\nInclude the following information in your report:\n- Type of vulnerability (e.g., SQL injection, XSS, etc.)\n- Affected versions\n- Steps to reproduce the issue\n- Potential impact\n- Any suggested fixes (if available)\n\n### Response Timeline\n\n- **Initial response**: Within 24 hours\n- **Detailed assessment**: Within 3 business days\n- **Resolution timeline**: Depends on severity, typically within 7-14 days\n- **Public disclosure**: After fix is deployed (with credit, if desired)\n\n### Security Measures\n\nThis project implements several security measures:\n\n#### Input Validation\n- All tool inputs are validated using Zod schemas\n- SQL injection prevention through parameterized queries\n- XSS protection in React widgets\n\n#### Database Security\n- Uses Prisma ORM with built-in SQL injection protection\n- Environment-based database credentials\n- SSL/TLS required for database connections\n\n#### API Security\n- No sensitive data in error messages\n- Rate limiting considerations for deployment\n- Environment variable protection\n\n#### Dependencies\n\nWe regularly update dependencies to patch known vulnerabilities:\n- Automated Dependabot updates\n- Regular security audits\n- npm audit for vulnerability scanning\n\n### Best Practices for Users\n\n#### Environment Variables\n\nNever commit `.env` files to version control. Always use `.env.example` as a template.\n\n#### Database Access\n\n- Use strong, unique passwords\n- Enable SSL/TLS connections\n- Regular database backups\n- Principle of least privilege for database users\n\n#### Deployment\n\n- Keep Node.js and dependencies updated\n- Use HTTPS in production\n- Monitor for suspicious activity\n- Regular security updates\n\n### Security Features\n\n#### Authentication & Authorization\n\nWhile the MCP server itself doesn't implement authentication (handled by MCP-Use platform), ensure:\n- Proper authentication on the deployment platform\n- Secure API key management\n- Access control for sensitive operations\n\n#### Data Protection\n\n- No sensitive data logged in production\n- Secure handling of wallet addresses\n- PII protection where applicable\n\n#### Monitoring\n\n- Implement logging for security events\n- Monitor for unusual API usage patterns\n- Set up alerts for potential security incidents\n\n## Security Advisories\n\n### Past Vulnerabilities\n\n*None reported to date.*\n\n### Known Security Considerations\n\n1. **Database Exposure**: Ensure DATABASE_URL is kept secret and uses SSL\n2. **CORS Configuration**: Review CORS settings for your deployment\n3. **Resource Limits**: Consider implementing rate limiting for production use\n4. **Widget Security**: React widgets should sanitize all user inputs\n\n## Responsible Disclosure Program\n\nWe believe in responsible disclosure and will:\n- Respond promptly to security reports\n- Work with researchers to understand and validate issues\n- Provide credit in our security advisories (with permission)\n- Strive to fix issues quickly\n\n## Security Team\n\nSecurity issues are handled by the ChainLoyalty development team:\n- Lead Developer: [Contact via security@chainloyalty.com]\n- Security Reviewer: [Contact via security@chainloyalty.com]\n\n## Additional Resources\n\n- [Node.js Security Best
